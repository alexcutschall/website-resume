Why did I add this?

If you've been working within with code for a while or have worked around programmers, none of these philosophies will surprise you much. They are ones that the experts within my field have mentioned. Nonetheless, I think the principals that drives each programmer are important. As I have continually strived to be a better programmer, I have accumulated certain philosophies that are central to how I architect and implement code.

Brass Tacks

  1. Testing: This deserves to be number one because how important I believe it is. This can be a divisive discussion within the programming community, but I've experienced first hand how much they impact code. They improve your design, they squash bugs before they enter QA, and they let you know if you break anything down the road.

  2. Clean Code: The influential book "Clean Code" by Uncle Bob is worth its weight in gold. It has impacted my code significantly and helps me make better coding decisions daily. Most of our time as programmers is digging through code, therefore we should try and do all we can to make it readable to our future selves and others.

  3. Design Decisions: This is a broad topic, but ultimately its about how we come to make design decisions. In my experience, having multiple design discussions with different architects is key to making well informed decisions for your code base. Design patterns help out immensely with this as it provides a common language for everyone to express themselves. A lot of this comes down to good communication, because code and architecture are complex and difficult topics, therefore the more we can find common ground, the better our decisions will be.
